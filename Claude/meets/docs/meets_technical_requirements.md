## 7. アーキテクチャ概要

### 7.1 システム構成図 (高レベル)

```
[クライアント (Browser)]
    |
    | HTTPS (TLS 1.2+)
    |
[CloudFront (CDN)] --- [S3 (画像ストレージ)]
    |
[ALB (Application Load Balancer)]
    |
    +-- [EC2 (Web/API Server) × N台]
          |
          +-- [Next.js (Frontend SSR)]
          +-- [Node.js + Express/Nest.js (Backend API)]
          |
          +-- [ElastiCache Redis (Session Store)]
          +-- [RDS PostgreSQL (Primary DB)]
          |
          +-- [AWS KMS (暗号化キー管理)]
          +-- [CloudWatch Logs/Metrics/Alarms]
          +-- [SNS → Slack (監視アラート)]

[外部API]
    +-- LINE Messaging API (通知)
    +-- Zoom API (ビデオ通話)
    +-- Stripe API (決済・送金)
    +-- eKYC SaaS (本人確認)
```

### 7.2 技術スタック (推奨)

| レイヤー | 技術 | 理由 |
|---------|------|------|
| **フロントエンド** | Next.js (React) + TypeScript | SSR/SSG対応、開発速度、TypeScriptによる型安全性 |
| **バックエンド** | Node.js + Express または Nest.js + TypeScript | フロントと言語統一、非同期処理に強い、豊富なライブラリ |

## 補足: 非機能要件の優先度付けロジック

本ドキュメントでは、以下の基準で非機能要件の優先度を付けている:

1. **Must (MVP必須)**: MVP公開前に必ず実装・達成する必要がある要件
   - セキュリティ: 全項目 Must (情報漏洩は事業リスク)
   - 信頼性: SLA、バックアップ、RTO/RPO は Must (サービス停止は致命的)
   - 性能: ページ読み込み3秒、検索2秒は Must (UX の最低ライン)

2. **Should (MVP推奨)**: MVP公開前に実装することが望ましいが、リソース不足時は Phase 2 に延期可能
   - 性能: API応答時間200ms、同時接続数100名
   - 可観測性: メトリクス収集、デバッグ用ログレベル切り替え
   - 運用: CI/CD、ロールバック手順、障害対応手順書

3. **Could (Phase 2以降)**: MVP では優先度が低く、Phase 2 以降で実装を検討
   - 拡張性: キャッシュ戦略
   - セキュリティ: ペネトレーションテスト
   - コンプライアンス: GDPR対応準備
