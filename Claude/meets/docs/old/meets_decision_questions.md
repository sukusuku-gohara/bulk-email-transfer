# meets 要件定義のための意思決定質問リスト

**作成日**: 2026-02-21
**対象**: エージェント型マッチングプラットフォーム「meets」MVP開発（2月〜4月）
**目的**: 要件定義を具体化し、受け入れ基準を設定するために必要な意思決定事項を明確化する

---

## 1. 業務要件カテゴリ

| # | 質問 | なぜ必要か（要件定義への影響） | 重要度 | 暫定方針（仮定案） |
|---|------|---------------------------|--------|-----------------|
| B-1 | 「成婚」の定義は何か？（真剣交際開始時点/プロポーズ成功時点/婚姻届提出時点） | 成婚料の発生タイミング・金額計算ロジック・CRMステータス遷移の設計に直結 | 高 | 「プロポーズ成功＋双方の成婚報告」を成婚とし、報告から7日以内に成婚料決済フローを開始する |
| B-2 | 成婚料は「ユーザーが支払う」のか「エージェントが設定した金額をユーザーが支払い、meetsが手数料を差し引く」のか？ | 決済フロー・Stripe連携・請求書発行・売上計上の設計が変わる | 高 | エージェントが成婚料を設定（例：20万円）→ユーザーが支払い→meetsが10%手数料を差し引き→エージェントに90%を送金 |
| B-3 | 月額会費はユーザーが支払うのか、エージェントが設定するのか？エージェント不在時の会費はいくらか？ | 決済システム・料金表示・会員ステータス管理の設計に影響 | 高 | エージェント紹介時：エージェントが月額会費を自由設定（例：5,000円〜30,000円）、エージェント不在時：meets側が一律10,000円/月に設定 |
| B-4 | 「お見合い」と「仮交際」と「真剣交際」のステータス遷移ルールは？誰が承認するのか？ | CRMパイプライン・ステータス管理・通知設計・エージェント業務フローの設計に必須 | 高 | お見合い→双方OK→仮交際（複数人同時可）→双方OK→真剣交際（1対1専念）→プロポーズ→成婚。ステータス変更は「ユーザーの申し込み＋相手の承認」で自動遷移、エージェントは助言のみ |
| B-5 | エージェント未登録ユーザーは「申請可能だが最終調整はエージェント介入必須」とあるが、エージェントが見つからない場合の対応は？また、未登録ユーザーの月間申請回数制限は？ | エージェントマッチングロジック・待機フロー・タイムアウト処理・申請回数制限の設計が必要 | 中 | 【エージェント不在時】申請から72時間以内にエージェントが見つからない場合、meets運営が暫定エージェントとして介入し、手数料10%をmeets側が徴収。【回数制限】エージェント未所属ユーザーは月間10回まで申請可能。エージェント所属ユーザーは制限なし。上限到達時にエージェント所属への誘導を実施（CHG-20260226-001で確定） |
| B-6 | アンバサダー紹介コードによるキックバック率は何%か？複数アンバサダー経由時の配分ルールは？ | レベニューシェア計算エンジン・支払い処理・不正検知の設計に必須 | 中 | アンバサダー紹介時：入会金・月額会費の5%をキックバック。複数経由時は「最初に登録したコード」を優先し、後続は無効とする |
| B-7 | NG判定（お断りリスト）は「双方向」か「片方向」か？NG登録後に再度表示されることはあるか？ | 検索除外ロジック・データベース設計・UXフローの設計に影響 | 中 | 片方向NG（AがBをNG登録→Aの検索からBが永久除外、BからはAが見える）。再表示機能は提供しない |
| B-8 | エージェントの「5段階評価」は誰が行うのか？（ユーザーのみ/運営も評価/相互評価） | 評価システム・表示ロジック・エージェントランキングの設計に影響 | 中 | ユーザーのみが評価可能。成婚時に自動で評価依頼通知を送信。評価は匿名で公開、平均評価が4.0未満のエージェントには運営から改善要請を送る |
| B-9 | meetsアカデミーの「認定証発行」条件は？不合格時の再受験ルールは？認定有効期限は？ | テスト合格判定・認定証発行フロー・エージェント資格管理の設計に必須 | 中 | 全動画視聴＋テスト80点以上で認定証発行。不合格時は24時間後に再受験可能（回数制限なし）。認定有効期限は1年、更新時は簡易テストのみ |
| B-10 | 外部連盟データ参照（TMS/BIU/JBA等）は「API連携」か「手動CSV取り込み」か？リアルタイム参照か定期同期か？ | データ連携方式・同期頻度・データ整合性・法的確認事項の設計に影響 | 中 | Phase 1では手動CSV取り込み（週1回）。Phase 2以降でAPI連携を検討。法的許可は各連盟と個別契約が必要 |
| B-11 | ユーザー登録情報の「13項目以上」の具体的な必須項目は？任意項目は？ | 登録フォーム設計・バリデーション・プロフィール表示の設計に必須 | 高 | 必須：氏名/生年月日/性別/居住地/身長/体重/学歴/職業/年収/結婚歴/子供有無/喫煙/飲酒。任意：趣味/自己PR/希望条件 |
| B-12 | 「活動ログ・メモ」はエージェントのみが記録できるのか、ユーザーも記録できるのか？相互閲覧可能か？ | データアクセス権限・UI設計・プライバシー設計に影響 | 低 | エージェントのみが記録可能。ユーザーは閲覧のみ可能（エージェントが「共有」設定したメモのみ）。エージェント間では非共有 |
| B-13 | 個別請求・決済リンク発行は「都度発行」か「定期自動発行」か？未払い時の督促フローは？ | 決済システム・請求管理・督促自動化の設計に影響 | 中 | 月額会費は毎月1日に自動請求。成婚料は成婚報告後7日以内に個別リンク発行。未払い3日後に自動督促、14日後にアカウント停止 |
| B-14 | フィードバック収集は「都度任意」か「成婚時必須」か？フィードバック未提出時の制約は？ | フィードバックフロー・通知設計・データ分析の設計に影響 | 低 | お見合い後・仮交際後・成婚時に任意フィードバック依頼。成婚時のみ必須（未提出でも成婚処理は完了するが、エージェント評価は依頼し続ける） |
| B-15 | 月間申請回数のリセットタイミングは暦月1日か、ユーザー登録日基準か？（CHG-20260226-001関連） | 申請回数管理ロジック・ユーザー体験の設計に影響 | 中 | 【暫定方針】暦月1日AM 0:00にリセット [ASSUMPTION]。登録日基準の場合、ユーザーごとにリセットタイミングが異なり管理が複雑化するため、暦月基準を推奨（Q-016で確認中） |
| B-16 | エージェント所属ユーザーの申請回数上限は設けないのか、別途上限を設けるのか？（CHG-20260226-001関連） | 申請回数管理ロジック・エージェント価値の設計に影響 | 中 | 【暫定方針】制限なし [ASSUMPTION]。エージェント所属の価値を明確化するため、制限なしを推奨。ただし、将来的に別途上限（例：月50回）を設定する可能性もあり（Q-017で確認中） |
| B-17 | 上限到達前の警告表示タイミングは何回目からか？（CHG-20260226-001関連） | UI表示・ユーザー体験の設計に影響 | 低 | 【暫定方針】8回目以降「残りN回」警告表示 [ASSUMPTION]。10回目到達時は案内画面を表示（Q-018で確認中） |
| B-18 | 月中にエージェント所属から未所属に戻った場合、当月の申請カウントはリセットか引き継ぎか？（CHG-20260226-001関連） | 申請回数管理ロジック・ユーザー体験の設計に影響 | 低 | 【暫定方針】引き継ぎ [ASSUMPTION]。未所属に戻った時点から月間10回制限が適用。当月の残り回数は所属前使用分を差し引き（Q-019で確認中） |
