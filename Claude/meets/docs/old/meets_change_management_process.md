# meets 仕様変更管理プロセスガイド

**作成日**: 2026-02-26
**バージョン**: 1.0
**管理者**: PM Agent (pm-agent)

---

## 1. 概要

本ドキュメントは、meetsプロジェクトにおける仕様変更の管理プロセスを定義する。全てのエージェントと関係者が本プロセスに従い、仕様変更を一貫した方法で記録・伝達・反映する。

### 1.1 目的

1. 仕様変更の発生から各ドキュメントへの反映までを追跡可能にする
2. 変更による影響範囲を漏れなく特定し、ドキュメント間の整合性を維持する
3. 各エージェントが自身の担当領域に影響する変更を確実に把握できるようにする

### 1.2 適用範囲

以下の全てに適用する。

- 要件の追加・変更・削除
- 未決事項の確定
- 制約条件の変更
- スコープの変更（MVP / Phase 2 / スコープ外の移動）
- 前提条件・仮定の変更

以下は適用外とする。

- 誤字脱字の修正
- フォーマットの統一・整形
- ドキュメント構造のリファクタリング（内容に変更がない場合）

## 2. 変更管理フロー

### 2.1 全体フロー図

```
[変更の発生]
    |
    v
[STEP 1] 変更の識別と起票 (pm-agent / 起票者)
    |
    v
[STEP 2] 影響分析 (pm-agent)
    |
    v
[STEP 3] 変更レコードの完成 (pm-agent)
    |
    v
[STEP 4] ドキュメント反映 (各担当エージェント)
    |
    v
[STEP 5] 整合性検証 (spec-writer-agent)
    |
    v
[完了]
```

### 2.2 各ステップの詳細

#### STEP 1: 変更の識別と起票

**担当**: pm-agent（又は変更を最初に認識したエージェント）

**トリガー**:
- ステークホルダーからの新しい指示・回答
- 会議での決定事項
- 技術検証による仕様見直し
- 競合分析結果による方針変更

**アクション**:
1. `meets_change_log.md` を開く
2. 変更テンプレートをコピーする
3. 以下を記入する:
   - 変更ID（`CHG-YYYYMMDD-NNN` 形式）
   - 変更日、起票者、変更種別、カテゴリ
   - 変更前の仕様、変更後の仕様、変更理由
4. ステータスを `起票` に設定する
5. 変更サマリーダッシュボードの件数を更新する
6. フィルタ用インデックスに行を追加する

#### STEP 2: 影響分析

**担当**: pm-agent

**アクション**:
1. ステータスを `影響分析中` に変更する
2. 以下の3つの影響領域を特定する:

**a) 影響を受ける要件IDの特定**

既存の要件ID体系から、変更によって影響を受けるIDを全て洗い出す。

| ID接頭辞 | 対象ドキュメント | 内容 |
|---------|----------------|------|
| D-NN | meets_structured_extraction.md | 決定事項 |
| U-NN | meets_structured_extraction.md | 未決事項 |
| BR-NNN | meets_business_requirements.md | 業務要件 |
| TR-NN | meets_structured_extraction.md | 技術要件候補 |
| UX-NN | meets_structured_extraction.md | UX要件候補 |
| OP-NN | meets_structured_extraction.md | 運用要件候補 |
| CMP-NN | meets_structured_extraction.md | コンプライアンス要件候補 |
| C-NN | meets_structured_extraction.md / meets_business_requirements.md | 制約条件 |
| RK-NN | meets_structured_extraction.md | リスク候補 |
| UNC-NN | meets_structured_extraction.md | 不明点 |
| NFR-NNN | meets_business_requirements.md | 非機能要件 |
| PR-NN | meets_structured_extraction.md | 前提条件 |
| ASM-NNN | meets_business_requirements.md | 仮定 |
| Q-NNN | meets_business_requirements.md | 確認質問 |
| US-NNN | meets_business_requirements.md | ユーザーストーリー |
| EC-NNN | meets_business_requirements.md | 例外ケース |
| AC-NNN-N | meets_business_requirements.md | 受け入れ基準 |

**b) 影響を受けるドキュメントの特定**

変更カテゴリに応じて、以下のドキュメントへの影響を評価する。

| 変更カテゴリ | 必ず確認するドキュメント |
|-------------|----------------------|
| 業務 | meets_business_requirements.md, meets_structured_extraction.md, meets_decision_questions.md |
| 技術 | meets_structured_extraction.md, meets_business_requirements.md (NFR) |
| UX | meets_business_requirements.md, meets_structured_extraction.md |
| 運用 | meets_business_requirements.md, meets_structured_extraction.md |
| コンプライアンス | meets_business_requirements.md, meets_structured_extraction.md |

**c) 担当エージェントの特定**

エージェント別影響マトリクス（meets_change_log.md 参照）に基づき、各ドキュメントの更新担当エージェントを特定する。

#### STEP 3: 変更レコードの完成

**担当**: pm-agent

**アクション**:
1. STEP 2 の結果を変更レコードの「影響を受ける要件ID」「影響を受けるドキュメント」テーブルに記入する
2. 各項目の「対応状況」を `未対応` に設定する
3. ステータスを `反映中` に変更する

#### STEP 4: ドキュメント反映

**担当**: 各担当エージェント

**アクション**:

各エージェントは、変更レコードの「影響を受けるドキュメント」テーブルで自身が担当に指定されたドキュメントを更新する。

**エージェント別の反映手順**:

| エージェント | 反映手順 |
|-------------|---------|
| **extractor-agent** | 1. meets_structured_extraction.md の該当テーブルを更新する 2. 変更前の記述は削除せず、備考欄に「CHG-YYYYMMDD-NNN により変更」と注記する 3. 変更レコードの対応状況を `反映済み` に更新する |
| **business-requirements-agent** | 1. meets_business_requirements.md の該当要件を更新する 2. 新規要件の場合は次の空き番号で BR-NNN を採番する 3. 削除の場合は要件を残したまま「スコープ外」列に移動する 4. meets_decision_questions.md の該当質問の回答を更新する 5. 変更レコードの対応状況を `反映済み` に更新する |
| **spec-writer-agent** | 1. 用語の整合性を確認する 2. 変更が要件定義書全体の構造に影響する場合、セクション再編を提案する 3. 変更レコードの対応状況を `反映済み` に更新する |
| **technical-agent** | 1. 非機能要件（NFR-NNN）への影響を評価する 2. 技術的実現性に問題がある場合、変更レコードの備考に記載する 3. 変更レコードの対応状況を `反映済み` に更新する |
| **benchmark-agent** | 1. 変更が競合ポジショニングに影響する場合、分析結果を備考に記載する 2. 変更レコードの対応状況を `反映済み` に更新する |

#### STEP 5: 整合性検証

**担当**: spec-writer-agent

**アクション**:
1. 全てのドキュメントの対応状況が `反映済み` になったことを確認する
2. ドキュメント間の整合性をチェックする:
   - 同一要件が複数ドキュメントに記載されている場合、内容が一致するか
   - 要件IDの参照が壊れていないか
   - 用語が統一されているか
3. 問題がなければ、変更レコードのステータスを `完了` に変更する
4. 問題がある場合、備考に不整合の内容を記載し、該当エージェントに差し戻す

## 3. 変更の起票タイミング

### 3.1 必ず起票するケース

| トリガー | 例 | 起票者 |
|---------|-----|--------|
| ステークホルダーからの質問回答 | 室井氏から「NG判定は双方向」との回答 | 回答を受け取ったエージェント |
| 会議での方針決定 | 「ビデオ通話はZoom連携に決定」 | 会議内容を抽出したextractor-agent |
| 技術検証による見直し | 「LINE API制限によりPush通知の頻度を制限」 | technical-agent |
| 法務確認結果 | 「成婚料の消費税扱いが変更」 | business-requirements-agent |
| 競合動向による方針変更 | 「競合が同等機能をリリースしたため差別化ポイントを変更」 | benchmark-agent |
| 仮定の検証結果 | 「ASM-001のeKYC処理件数が想定を下回る」 | technical-agent |

### 3.2 起票不要のケース

| ケース | 理由 |
|--------|------|
| 誤字脱字の修正 | 仕様内容に変更がない |
| Markdownフォーマットの調整 | 仕様内容に変更がない |
| 説明文の言い回し変更（意味変更なし） | 仕様内容に変更がない |
| 同一ドキュメント内の構造リファクタリング | 仕様内容に変更がない |

## 4. エージェントへの伝達方法

### 4.1 伝達フロー

仕様変更が発生した際、各エージェントへの伝達は以下の方法で行う。

**方法1: 変更ログの直接参照（標準）**

各エージェントは作業開始時に `meets_change_log.md` のフィルタ用インデックスを確認し、自身に関連する未対応の変更がないかチェックする。

**方法2: pm-agentからの指示（大規模変更時）**

影響度「高」の変更が発生した場合、pm-agentが各エージェントに以下の形式で指示を出す。

```
[仕様変更通知]
変更ID: CHG-YYYYMMDD-NNN
概要: [変更の概要]
あなたの担当:
  - [ドキュメント名]: [更新対象セクション] を [具体的なアクション]
参照: meets_change_log.md の該当レコード
```

**方法3: 新規入力処理時の自動反映**

extractor-agent が新しいメモ・議事録を処理する際、抽出結果に仕様変更が含まれていた場合:
1. 抽出結果を出力する
2. meets_change_log.md に変更レコードを起票する
3. pm-agent に影響分析を依頼する

### 4.2 各エージェントの作業開始時チェックリスト

```
[ ] meets_change_log.md のフィルタ用インデックスを確認した
[ ] 自身が担当するドキュメントに未対応の変更がないことを確認した
[ ] 未対応の変更がある場合、反映作業を優先実施した
```

## 5. ドキュメント間の整合性維持

### 5.1 ドキュメント間の関係性

```
[原本ソース]
  meets MVP開発検討資料.docx
  エージェント型マッチングプラットフォームmeets事業.pdf
      |
      v
[構造化抽出] (extractor-agent)
  meets_structured_extraction.md
  - D-NN: 決定事項
  - U-NN: 未決事項
  - PR-NN: 前提条件
  - C-NN: 制約条件
  - BR-NN (候補): 業務要件候補
  - TR-NN: 技術要件候補
  - UX-NN: UX要件候補
  - OP-NN: 運用要件候補
  - CMP-NN: コンプライアンス要件候補
  - RK-NN: リスク候補
  - UNC-NN: 不明点
      |
      v
[意思決定質問] (business-requirements-agent)
  meets_decision_questions.md / .xlsx
  - B-NN: 業務要件の質問
  - U-NN: UX要件の質問
  - T-NN: 技術要件の質問
  - O-NN: 運用要件の質問
  - S-NN: セキュリティ・コンプライアンスの質問
  - L-NN: 法務・ビジネスの質問
      |
      v
[業務要件定義書] (business-requirements-agent)
  meets_business_requirements.md / .xlsx
  - BR-NNN: 業務要件（確定版）
  - NFR-NNN: 非機能要件
  - US-NNN: ユーザーストーリー
  - AC-NNN-N: 受け入れ基準
  - EC-NNN: 例外ケース
  - ASM-NNN: 仮定
  - Q-NNN: 確認質問
      |
      v
[仕様変更履歴] (pm-agent)
  meets_change_log.md
  - CHG-YYYYMMDD-NNN: 変更レコード
```

### 5.2 ID体系の一貫性ルール

- 要件IDは**削除しない**。スコープ外になった場合もIDは保持し、ステータスのみ変更する
- 新規要件は常に次の連番で採番する（欠番は埋めない）
- 要件IDの変更（リナンバリング）は原則禁止。やむを得ない場合はmeets_change_log.md に対応表を記載する

### 5.3 整合性チェック項目

spec-writer-agent が検証時に確認する項目:

| チェック項目 | 確認方法 |
|-------------|---------|
| 要件IDの重複がないか | 全ドキュメントのID一覧を突合 |
| 同一要件が複数ドキュメントで矛盾していないか | 変更対象の要件IDを全ドキュメントで検索し、記述を比較 |
| 削除された要件がまだ参照されていないか | 削除対象IDで全文検索 |
| 新規要件の採番が正しいか | 最終番号からの連番であることを確認 |
| 変更履歴の記載が正確か | meets_business_requirements.md の 12. 変更履歴 セクションとmeets_change_log.md の整合性 |

## 6. 変更種別の定義と対応方針

| 変更種別 | 定義 | 対応方針 |
|---------|------|---------|
| **仕様追加** | 新しい要件・機能・制約の追加 | 新規IDを採番。既存要件との依存関係を確認。スコープ判定を実施 |
| **仕様変更** | 既存要件の内容変更 | 変更前後を明記。影響する受け入れ基準・例外ケースも更新 |
| **仕様削除** | 既存要件のスコープ外化 | 要件IDは保持しステータスのみ変更。スコープ外セクションに移動理由を記載 |
| **仕様確定** | 未決事項（U-NN）の確定 | 対応するU-NNをクローズ。確定内容を正式な要件ID（BR-NNN等）として定義 |

## 7. Excel (XLSX) ファイルとの同期

### 7.1 同期方針

Markdown (.md) ファイルを**正本 (Single Source of Truth)** とする。理由:

- 全エージェントがMarkdownを直接読み書きできる
- 差分の追跡が容易（git diff等）
- テンプレートの再利用が容易

Excel (.xlsx) ファイルは以下の目的で**副本**として維持する:

- ステークホルダー（経営陣等）への共有・レビュー用
- フィルタリング・ソート等の表計算操作が必要な場合
- 印刷・配布用

### 7.2 同期タイミング

- Markdownファイルに変更が反映された後、必要に応じてExcelファイルも更新する
- Excel更新の頻度は週1回（又はステークホルダーレビュー前）を目安とする
- **Excelファイル上での直接編集は禁止**。必ずMarkdown側を先に更新する

## 8. 変更管理のベストプラクティス

### 8.1 推奨事項

1. **変更は小さく記録する**: 1つの変更レコードに複数の無関係な変更を詰め込まない
2. **変更理由を必ず記録する**: 「誰が」「なぜ」変更したかが後から追跡できるようにする
3. **影響分析は過剰なくらいが適切**: 影響を見落とすよりも、不要な項目を挙げる方が安全
4. **変更前の仕様を正確に記録する**: 変更後だけでなく、何が変わったのかを明確にする
5. **ステータスを即座に更新する**: 作業状況の可視性を維持する

### 8.2 禁止事項

1. **meets_change_log.md に記録せずにドキュメントを変更する**（軽微なフォーマット修正を除く）
2. **変更レコードの削除**（誤記入の場合はステータスを「取消」に変更する）
3. **他エージェントの担当ドキュメントを無断で変更する**
4. **要件IDのリナンバリング**（原則禁止）
