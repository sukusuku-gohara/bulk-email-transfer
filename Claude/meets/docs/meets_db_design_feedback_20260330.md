## 全体評価

全体として、仕様の主要機能をよくカバーした設計である。ユーザー・プロフィール・お見合い・交際・チャット・決済・仲人・アンバサダーの各ドメインが適切に分離されており、正規化も妥当なレベル。ただし、最新仕様との突き合わせでいくつかの**不足・不整合**がある。

## 1. 不整合・修正が必要な項目

### 1-1. bookmarksテーブル: ソフトデリート vs 物理削除の矛盾

| 項目 | 内容 |
|------|------|
| **仕様** | TR-079「削除はソフトデリートではなく**物理削除**とする」と明記 |
| **DB設計** | `会員お気に入り`テーブルに`削除日時`カラムがあり、ソフトデリート設計になっている |
| **対応** | `削除日時`カラムを削除し、DELETE時は物理削除にする |
| **該当要件** | TR-079, BR-079 |

### 1-2. bookmarksテーブル: `created_by_role`カラムの欠如

| 項目 | 内容 |
|------|------|
| **仕様** | TR-079で `created_by_role` カラムを明示的に要求（カラム定義: id, user_id, target_user_id, created_by_role, created_at） |
| **DB設計** | 代わりに`ラベル`カラムがあるが、目的が異なる可能性が高い |
| **対応** | `created_by_role`（会員/エージェント）を追加。`ラベル`の必要性を要確認 |
| **該当要件** | TR-079 |

### 1-3. bookmarksテーブル: ユニーク制約の確認

| 項目 | 内容 |
|------|------|
| **仕様** | TR-079で `user_id + target_user_id` の複合ユニーク制約を要求し、重複ブックマークを防止 |
| **DB設計** | ER図では制約が読み取れない |
| **対応** | マイグレーション時に複合ユニーク制約を必ず付与すること |
| **該当要件** | TR-079 |

## 4. 設計上の注意・推奨事項

### 4-1. インデックス設計（TR-011対応）

`ユーザー.最終ログイン日時`と`会員プロフィール.created_at`に単体インデックス、および`(最終ログイン日時 DESC, created_at DESC)`の複合インデックスが必要（TR-011/TR-078/TR-080）。NFR-102（p95 < 2.0秒）を維持するため、マイグレーション時に必ず追加すること。

**必要なインデックス一覧:**

| テーブル | カラム | インデックス種別 | 根拠 |
|---------|--------|----------------|------|
| ユーザー | 最終ログイン日時 | 単体 B-tree | TR-078, TR-080 |
| 会員プロフィール | created_at | 単体 B-tree | TR-080 |
| ユーザー | (最終ログイン日時 DESC, created_at DESC) | 複合 B-tree | TR-011（推奨ソート用） |
| 会員お気に入り | (登録者ユーザーID, 対象ユーザーID) | 複合ユニーク | TR-079（重複防止） |
| 会員お気に入り | 対象ユーザーID | 単体 B-tree | TR-012（被ブックマーク数集計用） |

### 4-2. 月次お見合い申込回数の管理方式

`お見合い申込`テーブルに`月次申込回数スナップショット`があるが、月次カウントを各レコードのスナップショットで管理するのは**整合性リスクがある**（申込キャンセル時の巻き戻し、並行リクエスト時の競合等）。

**代替案:**
- **案A**: 月初リセットの専用カウンターテーブルを新設する（`monthly_request_counts`: user_id, year_month, count）
- **案B**: 毎回 `COUNT(*) WHERE 申込者 = ? AND 申込日時 BETWEEN 月初 AND 月末` でリアルタイム算出する（Phase 1の1,000名規模なら十分高速）
- **推奨**: 案B（Phase 1のスケールでは十分。カウンターテーブルの同期問題を回避できる）

### 4-3. `検索用プロフィール`カラムの用途

`会員プロフィール`にある`検索用プロフィール`（おそらくJSON型の非正規化フィールド）は、検索性能最適化として良いアプローチ。ただし、プロフィール更新時に同期が漏れないよう、更新トリガーまたはアプリケーション層でのatomicな更新を保証すること。

### 4-4. 振込テーブルのアンバサダー対応

`振込`テーブルは存在するが、成婚料の**エージェント・アンバサダーへの分配**（BR-040: エージェントとアンバサダーへの配分、meets 10%手数料）の計算結果をどのように保持するか確認が必要。

- 成婚1件に対して複数の振込レコード（エージェント分 + アンバサダー分）を生成する設計であれば問題ない
- `振込種別`カラムで agent_payout / ambassador_payout を区別できるか確認すること

### 4-5. RBAC（NFR-505）

`ユーザー.ユーザー種別`で会員/エージェント/アンバサダー/管理者を区別しており、MVPの3〜4ロール規模では十分。Phase 2以降で権限が細分化される場合は、roles/permissionsテーブルへの移行を検討。
