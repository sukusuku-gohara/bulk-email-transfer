## 抽出メモ（抽出結果の総評・注記）

### 入力品質・構造化度

- **資料 1（MVP 開発検討資料）**：機能仕様が詳細に記載されており、要件抽出の精度が高い。ただし、技術的な実装方式（Zoom vs 独自開発など）は複数オプションが並記され、決定が遅延している箇所が散見。

- **資料 2（事業企画書）**：経営・マーケティング観点の全体像は明確。ビジネスモデル・市場ポジショニング・成長戦略は説得力があるが、**実装タスクレベルでの詳細が不足**。特に「Phase 1/2/3」の移行トリガー・条件が曖昧。

### 抽出の傾向・パターン

1. **決定事項の明確さ**：プロダクト機能（会員向け・エージェント向け・アンバサダー向け）の大枠は決まっている。ただし、実装形式・優先度・スコープの境界線には曖昧さが残る。

2. **未決事項の頻出**：技術選定（Zoom vs 独自実装、データベース、インフラ）、外部連携仕様（連盟データ API）、計算ロジック（レベニューシェア）の 3 テーマが顕著に未決。

3. **前提条件の不完全性**：予算 5,000 万円は記載されているが、配分・実装優先度・人員計画が不明。開発体制（エンジニア数など）も明記なし。

4. **制約条件の法的側面**：eKYC・年齢確認・個人情報保護は記載されているが、APPI 準拠・GDPR 対応などの具体的なコンプライアンス要件は不足。

5. **曖昧な要件の多さ**：特に「成婚」「リアルタイム」「AI 仲人」などの定義が曖昧なまま使用されており、実装段階での混乱リスク。

6. **リスクの集中**：「エージェント 1,000 名獲得」「3 ヶ月での MVP 完成」「国際化対応」の 3 点が、ビジネス成立・スケジュール・スコープに大きく影響。

### 次フェーズへの推奨アクション

1. **2 月中に優先すべき確認事項**（要件定義固め）：
   - U-1（ビデオ通話の実装形式）、U-3（Chrome 拡張対象システム）、U-4（レベニューシェア計算式）の 3 点を確定
   - U-15（成婚定義）を明文化し、利用規約に反映
   - コンプライアンス・法務チームによる eKYC・個人情報保護の詳細設計

2. **要件定義書の作成時に明示すべき点**：
   - 「Phase 1」の機能スコープ・MVP に含まれない機能を明確に線引き
   - 各要件の「MUST/SHOULD/NICE-TO-HAVE」の優先度付け
   - 技術選定の根拠・判断基準の明記（例：「Zoom 連携を選定した理由：工期短縮・サポート負荷軽減」）

3. **不明点の所有者・解決時期の明確化**：
   - UNC-3（エージェント採用戦略）→ COO 室井 亘、1月中に施策立案
   - UNC-4（収益分岐点）→ CFO・経営企画、2月中に財務計画確定
   - UNC-13（インフラ詳細）→ CTO・システム設計チーム、2月中に設計決定

4. **リスク顕在化の早期発見**：
   - RK-3（ビデオ通話実装方式）：1月末までに技術検証・POC 実施
   - RK-2（エージェント 1,000 名獲得）：プロトタイプアカデミー・インセンティブ施策の早期テスト
   - RK-9（レベニューシェア計算）：設計完了後に単体テスト・複雑シナリオの検証

5. **ドキュメント整備**：
