# 一斉メール配信システム — 要件定義書

## 1. プロジェクト概要

自社の独自ドメインメールアドレス（Google Workspace連携）から、約1,000件の宛先に対して一斉メールを配信するシステムを自前サーバー上に構築する。

---

## 2. 前提条件

| 項目 | 内容 |
|------|------|
| 配信規模 | 約1,000件/回 |
| 配信頻度 | 週3回（月12回程度） |
| 送信元 | 独自ドメインのメールアドレス（Google Workspace紐づき） |
| メール送信方法 | Google Workspace SMTP経由 |
| メール内容 | 全宛先に同一内容（パーソナライズなし） |
| 配信リスト管理 | Googleスプレッドシート |
| インフラ | 自前サーバー（VPS） |

---

## 3. Google Workspace SMTP 送信上限に関する注意

Google Workspaceには1日あたりの送信上限がある。

- **Business Starter / Standard / Plus**：2,000件/日
- 週3回 × 1,000件 = 3,000件/週 → **1日1回の配信であれば上限内**
- 同日に2回配信すると上限を超える可能性があるため、**1日に複数回の配信は避ける運用ルール**が必要
- 上限を超えると、最大24時間メール送信がブロックされる

---

## 4. 機能一覧

### 4-1. メール配信機能

| 機能 | 説明 |
|------|------|
| 一斉送信 | 件名・本文（HTML対応）を指定し、リスト全体に配信 |
| スロットリング | 1秒あたりの送信数を制限し、サーバーやGmailに負荷をかけない（例：1秒に2〜3通） |
| テスト送信 | 本配信前に、自分宛にテストメールを送って内容を確認 |
| プレビュー | 送信前にブラウザ上でメールの見た目を確認 |
| 送信予約（任意） | 日時を指定して自動配信（Phase 2で検討可） |

### 4-2. 配信リスト管理機能

| 機能 | 説明 |
|------|------|
| スプレッドシート連携 | Google Sheets APIでアドレス一覧を取得 |
| アドレス追加・変更・削除 | スプレッドシート上で直接編集し、システムに反映 |
| 重複チェック | 同じアドレスが複数あった場合、自動で除外 |
| 配信停止（オプトアウト）管理 | 配信停止リクエストがあったアドレスを自動除外 |
| バウンス管理 | 届かなかったアドレス（存在しない等）を記録し、次回以降の配信から除外 |

### 4-3. 開封トラッキング機能

| 機能 | 説明 |
|------|------|
| トラッキングピクセル埋め込み | メール本文にリンク不要の小さな画像（1×1px）を自動挿入 |
| 開封記録 | 誰が・いつメールを開いたかをデータベースに記録 |
| 開封率表示 | 配信ごとの開封率を管理画面またはスプレッドシートで確認 |

**注意：** メールアプリが画像を自動ブロックしている場合、開封しても検知できない。実際の開封率は記録上の数値より高い。

### 4-4. 管理画面（Web UI）

| 画面 | 説明 |
|------|------|
| ダッシュボード | 直近の配信状況（件数・開封率）を一覧表示 |
| メール作成画面 | 件名・本文を入力し、プレビュー・テスト送信・本配信を実行 |
| 配信履歴 | 過去の配信一覧（日時・件数・開封率・バウンス数） |
| 開封詳細 | 配信ごとに、開封したアドレスの一覧を表示 |

## 5. 技術構成

| 要素 | 選定 | 備考 |
|------|------|------|
| サーバー | VPS（ConoHa / さくら等） | 月額1,000〜2,000円程度 |
| OS | Ubuntu 24.04 LTS | 長期サポート版 |
| 言語 / FW | PHP 8.3 / Laravel 13 | meetsプロジェクトと技術スタック統一 |
| DB | MySQL 8 | 配信ログ・開封ログの保存 |
| メール送信 | Google Workspace SMTP | ポート587 / TLS / アプリパスワード使用 |
| キュー処理 | Laravel Queue + Redis | 1,000件を順番に送る仕組み |
| リスト取得 | Google Sheets API v4 | サービスアカウントで認証 |
| トラッキング | 自サーバーでピクセル画像を配信 | Laravelルートでアクセスログを記録 |
| フロントエンド | Blade + Tailwind CSS | 管理画面用（シンプルなUI） |

## 6. 迷惑メール対策（必須）

一斉メールが迷惑メールと判定されないために、以下の設定が**必須**。

### 6-1. ドメイン認証

| 設定 | 説明 |
|------|------|
| SPF | 「このドメインからメールを送って良いサーバーはここです」とDNSに登録する仕組み |
| DKIM | メールに電子署名をつけて「改ざんされていない正規のメールです」と証明する仕組み |
| DMARC | SPFとDKIMの結果をもとに、不正メールをどう扱うか受信側に指示する仕組み |

Google Workspace経由の場合、SPFとDKIMはGoogle側で設定可能。DMARCは独自ドメインのDNSに追加する。

### 6-2. 配信停止リンク（2024年Gmail要件）

- メール本文内に**ワンクリックで配信停止できるリンク**を必ず含める
- メールヘッダーに `List-Unsubscribe` を設定する
- これがないと、Gmailが迷惑メールに分類するリスクが高まる

### 6-3. その他の対策

- 送信速度を制御する（一気に1,000件送らない）
- バウンスしたアドレスは速やかにリストから除外する
- メール本文に送信元の情報（社名・住所等）を含める

## 7. セキュリティ要件

| 項目 | 対応 |
|------|------|
| SMTP認証情報 | 環境変数（.env）で管理、コードに直書きしない |
| 管理画面アクセス | ログイン認証必須（Laravel Breeze等） |
| Google API認証 | サービスアカウントの秘密鍵を安全に保管 |
| HTTPS | 管理画面は常時SSL化 |
| 個人情報 | メールアドレスの暗号化保存を検討 |

## 10. 将来的な拡張候補（今回は対象外）

- パーソナライズ（名前の差し込みなど）
- 送信予約（日時指定で自動配信）
- クリックトラッキング（メール内リンクのクリック率計測）
- A/Bテスト（件名を2パターン送って効果比較）
- 配信リストのセグメント分け（グループ別配信）
- リスト件数が大幅に増えた場合の自前メールサーバー移行
